# `src/sentiment.py`

## Для какой задачи нужен файл

Этот файл отвечает за Table 4:

- анализ тональности отзывов IMDB.

## Почему этот файл самый сложный среди всех

Первые таблицы работают с сигналами и синтетическими временными рядами.
Table 4 работает с текстом.

Это резко меняет задачу:

- на входе уже не гладкий сигнал,
- а последовательность индексов слов,
- сначала эти индексы нужно превратить в осмысленные векторы,
- и только потом подавать в осцилляторную модель.

## Что делают основные части файла

### `make_ramp_targets(labels, num_steps, num_classes=2)`

Что делает:

- строит последовательностную цель из обычной метки класса.

Как именно:

- для правильного класса создаётся растущая рампа от `0` до `1`,
- для неправильного класса остаётся ноль.

Зачем:

- это приближает постановку к стилю статьи,
- где задача описывается как последовательностный выход, а не как один финальный логит.

### `labels_from_targets(y)`

Что делает:

- переводит последовательностные цели обратно в обычные метки классов.

### `aggregate_sequence_logits(logits)`

Что делает:

- суммирует выход модели по времени.

Зачем:

- итоговый класс берётся не по одному шагу,
- а по накопленному сигналу всей последовательности.

### `sequence_accuracy(logits, y_true)`

Что делает:

- считает accuracy уже после суммирования по времени.

### `split_train_val(...)`

Разделяет train и validation.

### `_select_subset(...)`

Что делает:

- позволяет брать только часть IMDB-датасета.

Почему это полезно:

- полный IMDB с DONN на CPU был бы слишком тяжёлым для быстрых showable-прогонов.

### `load_imdb_dataset(...)`

Это входная точка для данных.

Что делает:

1. грузит IMDB из `keras.datasets.imdb`,
2. ограничивает словарь top-35000,
3. опционально оставляет только подмножество train/test,
4. дополняет или обрезает отзывы до длины 500.

### `get_imdb_decoder()` и `decode_review(...)`

Зачем нужны:

- чтобы в визуальном отчёте показывать не просто числа токенов,
- а человекочитаемый текст.

Важно:

- сами отзывы остаются на английском,
- потому что исходный датасет IMDB англоязычный.

### `DONNSentimentClassifier`

Это основная DONN-модель для текста.

Её схема:

1. `Embedding`,
2. первый Hopf-блок,
3. `ReLU`-проекция,
4. второй Hopf-блок,
5. ещё одна `ReLU`-проекция,
6. компактный `tanh`-слой,
7. последовательностный выход на 2 канала.

Почему здесь два Hopf-слоя:

- так ближе к описанию статьи для Table 4.

Что важно:

- частоты осцилляторов здесь обучаемые,
- это специально поддержано в `HopfLayer`.

### `SentimentMetrics`

Хранит:

- `test_acc`
- `val_acc`
- `test_loss`

### `train_one_run(...)`

Это основной training pipeline DONN для Table 4.

Что делает:

1. грузит данные,
2. строит рамповые цели,
3. делит train на train/val,
4. создаёт модель,
5. учит её по MSE,
6. считает accuracy через суммирование по времени.

Почему это важно:

- именно здесь мы увидели, что локальный перенос DONN на текст пока работает плохо.

### `train_bilstm_baseline(...)`

Это sanity-check baseline.

Что делает:

- строит обычную модель:
  - `Embedding`
  - `Bidirectional LSTM`
  - `Dense`
- учит её как стандартный классификатор.

Зачем:

- чтобы не спутать “DONN не справился” с “датасет или разбиение сломаны”.

Именно baseline показывает:

- сама постановка живая,
- проблема сидит в DONN-адаптации под текст.

## Главный вывод по файлу

Этот файл важен не только как реализация Table 4,
но и как диагностический инструмент.

Он честно показывает, что:

- локально paper-like параметры ещё не дают paper-like качество,
- на тексте DONN-перенос пока слабее простой LSTM-базы.

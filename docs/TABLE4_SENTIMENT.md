# Table 4: проверка точного варианта статьи

Этот файл нужен для двух разных сценариев Table 4:

1. `visual/sentiment_result.py`
это локальная удобная адаптация, которую проще запускать и разбирать;

2. `visual/sentiment_paper_result.py`
это более строгий контрольный прогон, который старается повторить именно то,
что написано в полной таблице статьи.

## Что прямо написано в статье

Из текста статьи и полной Table 4 следует такой DONN-вариант:

- датасет: IMDB sentiment analysis;
- длина отзыва: `500`;
- словарь: `35000`;
- embedding: `100`;
- архитектура:
  `Embedding(100) -> Hopf(100) -> ReLU(100) -> Hopf(100) -> ReLU(100) -> tanh(20) -> output(2)`;
- начальный диапазон частот: `1-15 Hz`;
- частоты осцилляторов обучаются;
- тип входа в осциллятор: `I(t)`;
- оптимизатор: `Adam`;
- learning rate: `0.001`;
- loss: `MSE`;
- split train/validation: `7:3`;
- заявленная точность DONN: `85.2%`.

## Что оказалось нестыковочным уже на уровне статьи

В тексте рядом с Table 4 также указано:

- `total trainable parameters = 26,798`.

Но при буквальном следовании архитектуре из полной таблицы возникает проблема:

- один только embedding на `35000 x 100` уже даёт `3,500,000` параметров.

Значит хотя бы одно из следующих утверждений в статье недоопределено или
записано неточно:

- число параметров;
- способ подсчёта параметров;
- реальная архитектура;
- способ использования embedding-слоя.

## Что именно делает контрольный runner

Файл `visual/sentiment_paper_result.py` использует отдельный путь обучения из
`src/sentiment.py`, чтобы не смешивать контрольный прогон со старой локальной
аппроксимацией.

Текущая интерпретация paper-style запуска такая:

- задача трактуется как many-to-one классификация;
- целевые значения это one-hot вектор размера `2`;
- loss остаётся `MSE`, как и в статье;
- после второго Hopf-блока берётся последний временной шаг;
- затем идут `tanh(20)` и `output(2)`.

Это не полностью доказывает, что именно так обучали авторы статьи, но это уже
гораздо ближе к полной Table 4, чем старый ramp-based вариант.

## Подтверждённые локальные результаты

### 1. Paper-style DONN, `1024 / 1024`, `1` эпоха

Файлы:

- `artifacts/plots/fourth_work_paper_exact_summary_1k1e.png`
- `artifacts/plots/fourth_work_paper_exact_metrics_1k1e.json`

Результат:

- `test_acc = 0.5254`
- `val_acc = 0.4886`
- `test_loss = 0.4255`
- `total_params = 3,542,462`
- `non_embedding_params = 42,462`

Для сравнения на том же типе сплита сохранён короткий baseline:

- `artifacts/plots/fourth_work_paper_baseline_metrics_1k1e.json`

Baseline:

- `test_acc = 0.5605`
- `val_acc = 0.5472`

### 2. Paper-style DONN, `4096 / 4096`, `3` эпохи

Файлы:

- `artifacts/plots/fourth_work_paper_exact_summary_4k3e.png`
- `artifacts/plots/fourth_work_paper_exact_metrics_4k3e.json`

Результат:

- `test_acc = 0.4846`
- `val_acc = 0.4898`
- `test_loss = 0.2513`

Важно:

- loss убывает;
- accuracy не выходит из области случайного угадывания.

Это сильный признак того, что paper-style MSE-постановка для Table 4 либо
описана в статье неполно, либо в опубликованной формулировке не воспроизводит
заявленный результат.

## Главный вывод

На текущий момент у нас есть уже не просто ощущение, а конкретные признаки, что
“точный вариант статьи” по Table 4 проблемный:

- полная Table 4 и текст статьи дают внутренне противоречивое число параметров;
- более буквальный control-run не приближается к `85.2%`;
- при росте числа примеров и эпох accuracy не стабилизируется;
- даже короткий baseline на тех же данных ведёт себя лучше.

Поэтому только после такого контрольного этапа имеет смысл переходить к
собственным улучшениям Table 4.
